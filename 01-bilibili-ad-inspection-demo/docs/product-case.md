# 产品案例：B站投放巡查系统 Demo

## 背景

在投放业务里，账户和创意数量一多，人工巡查会出现三个问题：

- 看不过来：多个账户、多个创意、多个指标同时变化。
- 判断不一致：不同投手对“异常”的阈值和处理节奏不同。
- 复盘困难：当天为什么预警、为什么进入关停候选、后续是否恢复，缺少统一记录。

这个 demo 把真实巡查流程整理成安全版本，用 synthetic 数据展示产品逻辑。

## 需求分析

### 用户痛点

| 痛点 | 业务影响 | 产品机会 |
|---|---|---|
| 空耗发现晚 | 预算被消耗但没有转化 | 建立零转化空耗规则 |
| CPA 波动大 | ROI 低，投放负责人难以及时调整 | 建立超成本和账户级阈值规则 |
| 低效素材继续跑 | 点击和互动有消耗，但承接弱 | 引入转化率、CPM、评论点击成本 |
| 余额不足未提前发现 | 账户突然断投 | 建立余额风险提醒 |
| 通知分散 | 运营人无法快速定位自己负责的账户 | 按运营人分组输出通知预览 |
| 处理记录分散 | 复盘时很难还原过程 | 输出结构化巡查报告 |

### MVP 目标

让外部评审者在 3 分钟内看到：

1. 系统读取什么数据。
2. 系统如何按真实流程判断风险。
3. 系统输出什么报告和通知预览。
4. 人下一步如何处理。

## 用户流程

```text
投放负责人准备指标数据
  -> 系统执行巡查
  -> 查看 P0/P1/P2 风险
  -> 按运营人收到通知预览
  -> 人工确认处理动作
  -> 复盘规则是否需要调整
```

## 指标设计

| 指标 | 说明 | 为什么重要 |
|---|---|---|
| cost | 当前消耗 | 判断预算是否已经产生风险 |
| conversions | 转化数 | 判断是否空耗 |
| CPA | cost / conversions | 判断获客成本是否过高 |
| ROI | 回收效率 | 判断整体投放质量 |
| conversion_rate | conversions / clicks | 判断点击到转化是否低效 |
| CPM | 千次展现成本 | 判断曝光成本是否异常 |
| comment_click_cost | 评论点击成本 | 判断互动成本是否异常 |
| cash | 账户现金余额 | 判断是否可能断投 |

## 巡查规则

| 风险级别 | 规则 | 建议动作 |
|---|---|---|
| P0 | 零转化空耗或命中 dry-run 关停候选 | 优先人工确认，必要时进入处理队列 |
| P1 | 超成本或余额不足 | 路由给对应运营人复核 |
| P2 | 低转化率或 ROI 下滑 | 进入观察和复盘列表 |
| 保护样本 | 命中 `skip_autopause=true` | 只告警，不进入自动关停候选 |

## 产品边界

公开 demo 不做自动真实关停。原因：

- 公开环境不应该连接真实平台权限。
- 自动关停必须依赖真实权限、白名单、保护期和审计记录。
- 公开 demo 只展示规则、报告、通知预览和 dry-run 候选。

## 验收标准

| 验收项 | 标准 |
|---|---|
| 可运行 | 一条命令生成巡查报告和通知预览 |
| 可理解 | README 能说明用户、痛点、流程和指标 |
| 可复查 | synthetic 数据、规则配置和报告能一一对应 |
| 可扩展 | 后续能接前端看板、规则后台或真实适配器 |
| 安全 | 不包含真实账号、真实数据、token、cookie、Webhook、日志 |

## 后续产品化方向

- 增加前端看板：风险列表、规则配置、趋势图。
- 增加操作审计：每次处理动作保留原因和结果。
- 增加规则版本：不同业务阶段使用不同阈值。
- 增加 AI 解释：把异常原因翻译成运营可读建议。
