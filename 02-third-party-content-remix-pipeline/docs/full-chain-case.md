# 产品案例：第三方内容对标改写全链路

## 背景

内容创作里最容易出现的问题是：看到一条好视频，就让 AI “仿写一条”。这种方式很快，但风险也很大：

- 容易搬运原句或原作者案例。
- 学不到内容结构，只得到一条短期文案。
- 没有沉淀到自己的知识库，下次还要重新拆。
- 缺少证据替换，内容不像自己真实经历。

这个 demo 把真实流程整理成公开脱敏版本，重点展示“采集、转写、拆解、沉淀、改写”的闭环。

## 用户流程

```text
选择第三方公开内容
  -> 检查是否值得深拆
  -> 获取转写或 metadata fallback
  -> 写样本准入卡
  -> 做七遍编导拆解
  -> 质量评分
  -> 沉淀复用资产
  -> 生成知识库同步计划
  -> 改写成自己的脚本包
```

## 为什么这比“直接仿写”更有价值

| 直接仿写 | 全链路拆解 |
|---|---|
| 只得到一条文案 | 得到结构、资产和脚本包 |
| 容易像原作者 | 替换成自己的项目证据 |
| 难以复盘 | 每一步都有 Markdown 产物 |
| 容易泄露平台原文 | 公开 demo 只保留 synthetic data |
| 没有长期积累 | 有知识库同步计划 |

## 七遍拆解的输出

| 轮次 | 问题 | 产物 |
|---|---|---|
| 1 | 样本是否值得深拆 | 样本准入卡 |
| 2 | 内容整体怎么推进 | 整体结构 |
| 3 | 每句话在做什么 | 逐句功能 |
| 4 | 用户心理怎么变化 | 心理路径 |
| 5 | 表达技巧是什么 | 钩子、句式、节奏 |
| 6 | 转化怎么发生 | CTA 和承接方式 |
| 7 | 哪些能复用 | 结构、句式、反例资产 |

## 知识库边界

公开 demo 不写入真实 Obsidian，只生成 `knowledge-base-sync-plan.md`。真实流程中，只有这些内容适合沉淀：

- 可迁移结构。
- 可替换句式模板。
- CTA 和转化边界。
- 反例和风险提醒。
- 与木易项目证据相关的创作方法。

这些内容不能沉淀：

- 原作者真实案例。
- 原视频原句。
- 真实评论、昵称、头像、联系方式。
- 未核验的结果承诺。

## 验收标准

- 一条命令能生成完整链路产物。
- 每个产物能对应真实流程的一个阶段。
- 改写脚本能看出自己的项目证据。
- 没有真实平台数据、凭证、评论原文或知识库路径。
