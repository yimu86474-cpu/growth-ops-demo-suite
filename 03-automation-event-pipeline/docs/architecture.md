# 架构说明：自动化事件流底座

## 核心问题

很多业务自动化一开始会写成“收到消息就直接执行动作”。这样风险很高：误触发、误发送、无法复盘、无法停止。

事件流底座的目标是把动作拆开，让每一步都可观察、可确认、可回放。

## 分层架构

| 层级 | 职责 | 公开 demo 中的表达 |
|---|---|---|
| 输入层 | 接收外部消息、表格、网页文本、人工输入 | 只用模拟数据 |
| 事件层 | 标准化、去重、写入队列 | 展示 JSON/CSV schema |
| 判断层 | 规则或 AI 判断任务类型 | 只输出草稿和建议 |
| 输出层 | 写报告、任务或待确认队列 | 不真实发送 |
| 审计层 | 保存处理记录和结果 | 用样例报告展示 |

## 风险控制

- 高风险动作默认进入 `needs_review`。
- 真实发送、真实关停、真实同步必须由人确认。
- 公开 demo 不保存运行日志和真实消息。
- 配置文件只提供 example，不提供 local。

## 和主作品的关系

B站投放巡查系统可以把巡查结果写成事件，进入预警队列。

AI 内容研究工作台可以把评论问题和项目过程写成素材事件，进入选题和脚本流程。

事件流底座让这些项目共享一套安全、可复盘的自动化思想。
