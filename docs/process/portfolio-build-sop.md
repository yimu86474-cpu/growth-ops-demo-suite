# 公开演示包构建 SOP

## 适用场景

当本地已经有多个真实项目，但它们包含真实账号、配置、日志、截图、逐字稿或运行过程资产时，不直接公开原仓库，而是创建一个 public-safe demo suite。

这份 SOP 给未来的木易和 Codex 复用：把私有项目整理成可查阅、可运行、可复核的公开演示包，用产品案例、synthetic 数据、体验流程和边界声明证明能力。

## 核心原则

公开仓库不是生产仓库的复制品，而是“脱敏后的产品表达层”。

外部评审者要看的不是原始数据和真实账号，而是：

- 识别了什么业务问题。
- 如何抽象用户、流程、指标和风险。
- 如何把真实流程整理成可体验 demo。
- 如何用文档、截图、视频和 Release 让别人快速理解。
- 是否知道哪些内容不能公开。

## 为什么不直接公开真实项目

真实项目通常包含四类不适合公开的东西：

- 本机运行条件：`.env`、`*.local.*`、token、cookie、key、Webhook、本地路径、浏览器 profile。
- 真实业务现场：账号 ID、人员姓名、客户信息、群聊、聊天原文、投放数据、订单数据、调试日志。
- 平台敏感痕迹：签名 URL、带鉴权参数的链接、截图视频里的头像/昵称/后台数据。
- 过程资产：runtime、output、逐字稿、临时缓存、失败日志、录屏素材。

更好的做法是把真实项目拆成三层：

| 层级 | 是否公开 | 作用 |
|---|---|---|
| 私有真实项目 | 不公开 | 保留真实运行逻辑、配置、历史现场和验证证据 |
| 脱敏公开演示包 | 公开 | 展示产品思路、demo、synthetic 数据、案例说明和边界 |
| 本地讲述材料 | 不进入 release | 只给自己准备，不放进公开包 |

## 三类工作分工

| 工作 | 核心职责 | 输入 | 输出 | 边界 |
|---|---|---|---|---|
| 执行 | 把真实项目整理成 demo 和案例 | 私有项目只读入口、用户目标、公开仓库 | demo、README、案例文档、synthetic 数据、截图/视频计划 | 不复制真实数据，不修改真实项目仓库 |
| 复查 | 审查公开安全和可读性 | 执行产物 | 风险清单、修改建议、发布结论 | 命中 P0 风险必须阻断发布 |
| 沉淀 | 把流程升级成可复用 SOP | 执行过程、复查结果、踩坑记录 | SOP、review checklist、下一轮迭代建议 | 只沉淀方法，不写入真实账号和运行现场 |

## 标准流程

### 1. 识别业务闭环

先用产品问题描述真实项目：

| 问题 | 需要回答 |
|---|---|
| 服务谁 | 目标用户是谁，外部读者能否一眼看懂 |
| 解决什么痛点 | 原来靠人工做什么，痛点在哪里 |
| 输入是什么 | 用户、文件、页面、消息、表格、公开内容还是模拟数据 |
| 系统怎么判断 | 规则、AI、流程、评分、审核、人机协作分别做什么 |
| 输出是什么 | 报告、看板、草稿、任务、脚本、视频、表格还是提醒 |
| 如何验收 | 指标、截图、demo、运行命令、边界声明是什么 |

产物：一段 `case brief`，不要超过 300 字，能放进 README。

### 2. 保留流程，替换敏感现场

抽象时保留能力，不保留身份痕迹：

- 保留流程、字段结构、判断规则、指标体系。
- 替换账号、人员、链接、平台签名、截图、历史文本。
- 把真实触发器改成 synthetic 输入或手动上传。
- 把真实发送/发布改成 dry-run、draft-only 或本地预览。

### 3. 制作 synthetic 数据

synthetic 数据必须像真实业务，但不能来自真实业务。

建议规则：

- 账号统一使用 `ACC-DEMO-001`、`CR-DEMO-001`、`Operator Demo A`。
- 金额、ROI、转化率等指标可以保持业务量级，但要重新生成。
- 评论、需求、聊天、逐字稿全部重新写，不从真实内容改几个字。
- 数据集中加入正常样本、异常样本、边界样本，方便 demo 展示判断能力。
- 在数据文件或 README 里明确写：`All demo data is synthetic.`

### 4. 写产品案例

每个 demo 建议用同一套结构：

```text
项目名称
一句话定位
业务背景
目标用户
核心流程
能力亮点
可体验 demo
截图或视频
运行说明
安全边界
下一步迭代
```

写法要像产品案例，不像代码备忘录：

- 先讲业务问题，再讲实现。
- 先讲闭环，再讲模块。
- 先讲读者能验证什么，再讲做了什么。
- 对 AI 能力写清楚“AI 负责什么，人负责什么”。

### 5. 做可体验 demo

demo 的最低标准：

- 不需要真实 token、cookie、账号登录。
- 使用 synthetic 数据能跑通一个完整闭环。
- 有明确入口，例如 `README` 的运行命令、网页入口或视频入口。
- 能展示输入、处理、输出，而不是只展示一堆文件。
- 默认 `dry-run`，任何发送、发布、上传、写入外部系统都必须关闭。

### 6. 安全扫描

发布前必须完成安全脱敏检查。重点看这些类别：

- token/cookie/key：`.env`、API key、access token、refresh token、session、Webhook。
- local config：`config.local.*`、`*.local.json`、本机路径、设备 ID、浏览器 profile。
- runtime/output：运行日志、缓存、截图、outbox、inbox、sent、reports、逐字稿、临时文件。
- 真实账号/人员/数据：账号 ID、昵称、姓名、手机号、邮箱、客户名、群名、真实消耗、真实转化。
- 截图视频：头像、昵称、后台字段、二维码、浏览器地址栏、插件信息、通知弹窗。
- 平台签名 URL：带 `token`、`sign`、`timestamp`、`xsec`、`session`、`auth` 等参数的链接。

### 7. 发布

发布不是只 push 代码，而是做成外部读者能快速验收的产品包。

发布前至少具备：

- 顶层 README：说明公开演示包定位和 demo 索引。
- 每个 demo 的案例页：业务背景、流程、demo、边界。
- 截图：展示关键界面或关键输出。
- Release：写清楚这一版可看什么、如何运行、边界是什么。
- 运行说明：最小可跑命令、依赖、synthetic 数据入口。
- 边界声明：明确不包含真实账号、真实数据和真实平台操作。

## 完成定义

一次公开演示包构建任务完成，必须同时满足：

- demo 能用 synthetic 数据跑通或通过视频完整展示。
- README 和案例页能让非技术读者理解业务价值。
- review checklist 没有 P0 安全问题。
- 发布材料包含截图、运行说明、边界声明和下一步迭代。
- 流程经验已更新到 SOP 或 checklist。
